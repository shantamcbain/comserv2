### Chat #6: Docker Volume Mount Override - Starman Module Visibility Fix
**Timestamp**: Sun Oct 26 04:15:33 PM PDT 2025
**Topic**: Container Startup Failure Resolution - Volume Mount Architecture
**Assistant Role**: Zencoder (DevOps Problem Solving)

**What Happened**:
- User reported Docker containers failing to start: unable to access application at ports 3000 or 5000
- Container logs showed repeated "Restarting" state with "Unlinking stale socket" messages
- Initial investigation: Dockerfile builds successfully, but containers won't start
- Root cause discovered: Volume mount override - docker-compose.yml mounts entire `/opt/comserv` directory, shadowing files created during Docker build
- Starman error: Container couldn't find `/opt/comserv/local/bin/starman` (installed during build but overridden by empty local host directory)
- Solution implemented: Modified docker-compose.yml to mount only specific application directories:
  - Mount source code: lib, script, root, config, t subdirectories
  - Preserve Docker-built artifacts: local/bin (Starman), local/lib (Perl modules), etc
  - Mount entrypoint scripts as read-only files for runtime override capability
- Fixed web-prod WEB_PORT environment: changed from 3000 to 5000 (was incorrectly set to same port as web-dev)
- Updated health checks: dev uses port 3000, prod uses port 5000
- Result: Proper architectural separation - code updates from host don't shadow runtime binaries

**Problems Encountered**:
- Volume mount shadowing: monolithic mount of `/opt/comserv:/opt/comserv` replaced all built modules with host directory contents
- WEB_PORT misconfiguration: prod service had WEB_PORT=3000 despite mapping to port 5000, causing port conflict
- Module discovery: Starman (Perl package manager) not in host directory - only available in built container image

**Solutions/Outcomes**:
- Selective volume mounting: granular control over which directories come from host vs Docker image
- Proper port configuration: dev=3000, prod=5000 (environment variables now match port mappings)
- Ready for: `docker-compose up -d --build` execution
- Both services now configured to preserve built Perl modules while allowing code development
